# Token Pipeline — Template Architecture

How to turn this repo into a reusable, per-client token pipeline.
Workflow: **Figma (source of truth) → Claude (development) → Code (production)**.

## The three layers

| Layer | What it is | Distribution | Changes per client? |
|-------|-----------|--------------|---------------------|
| **Shared core** | The build machinery — config, scripts, Storybook setup, agent rules | Versioned npm package (`@scope/token-pipeline-core`) **or** baked into the template at first | No — identical everywhere |
| **Template** | Per-client scaffold that wires the core to one Figma file | GitHub template repo / `degit` | Cloned once per client, then edited |
| **Per-client content** | The tokens themselves + generated outputs | Lives in each client repo (and optionally published as that client's own package) | Yes — unique brand |

Start with just the **template** layer. Once you've replicated 2–3 clients and seen
what's genuinely stable, extract the core into its own package so pipeline
improvements propagate via a version bump instead of copy-paste.

## File-by-file classification

### Shared core (machinery — reuse verbatim)

- `sd.config.mjs` — Style Dictionary build engine; reads `prefix` from `pipeline.config.mjs` (with a `TOKEN_PREFIX` env override), no hardcoded client values
- `scripts/snapshot-tokens.mjs` — changelog/snapshot diff; reads fixed `tokens/*` paths, fully generic
- `scripts/generate-docs.mjs` + `templates/agent-rules.md` — render `AGENTS.md`/`CLAUDE.md` from one source
- `scripts/scaffold-client.mjs` — one-command per-client setup (see below)
- `.storybook/main.js`, `.storybook/preview.js` — Storybook config
- `src/stories/*.jsx` — Colors / Typography / Spacing / BorderRadius / Changelog stories; render tokens generically
- `design.md`, `PROCESS.md` — the design rule + token-change workflow
- `devDependencies` and the npm `scripts` block in `package.json`

### Template (scaffold — set once via `scaffold-client`)

- `pipeline.config.mjs` — **the single source of per-client config**: `projectName`, `prefix`, `figmaFileName`, `figmaFileKey`
- `package.json` — name, scope, version, description, author, repository (set by the scaffold script)
- `.github/workflows/chromatic.yml` — already references a secret, so just set the secret (no edit)
- `.gitignore`, `README`
- `tokens/` left in place until the client's first Figma sync overwrites it

`CLAUDE.md` and `AGENTS.md` are **generated**, not edited — they come from
`templates/agent-rules.md` + `pipeline.config.mjs`, so they can never drift and
carry no hardcoded client values.

### Per-client content (generated or synced — never shared)

- `tokens/tokens.{light,dark}.json`, `tokens/color.json`, `tokens/typography.json`, `tokens/size.json`, `tokens/guidelines.json` — synced from that client's Figma
- `tokens/snapshot.json`, `tokens/changelog.json` — generated by `npm run build`
- `dist/{light,dark}/**` — generated outputs (`variables.css`, `tokens.js`, `tokens.flat.json`)

## Spinning up a new client — step-by-step runbook

A complete walkthrough, written to be followed even if you've never used a
terminal. You'll use the macOS **Terminal** app (find it with Spotlight: press
⌘-Space, type "Terminal", press Enter). In the boxes below, each line is one
command — copy it, paste into Terminal, press Enter, and let it finish before the
next. Lines starting with `#` are explanations, not commands; you don't type them.

> You can also skip the terminal entirely and ask Claude to do any step in plain
> English (e.g. *"copy my IDEM template into a new folder called acme-tokens"*).
> The commands below are simply what Claude runs under the hood — useful when you
> want to understand or do it yourself.

Throughout, replace **Acme** / **acme** with your client's real name, and
`AbC123` with their real Figma file key.

### Step 0 — Make a fresh copy of the template
Each client gets its own folder. Never work inside the original
`IDEM Token Pipeline` folder.

```bash
# 1. Go to the folder that holds your projects
cd ~/Documents/Claude/Projects

# 2. Copy the template into a new folder named after the client.
#    The three --exclude bits leave out parts that shouldn't be in a fresh copy.
rsync -a --exclude node_modules --exclude .git --exclude dist "IDEM Token Pipeline/" acme-tokens/

# 3. Move into the new folder. Every command after this runs from here.
cd acme-tokens
```

Why those three are excluded: `node_modules` is a large folder of downloaded
tools (rebuilt in Step 3); `.git` is the *template's* history (your client needs
its own — Step 5); `dist` is generated output (rebuilt in Step 3).

### Step 1 — Preview the scaffold (changes nothing yet)
The `--dry-run` flag shows what *would* happen without writing anything. Always do
this first and read the list it prints.

```bash
npm run scaffold-client -- \
  --name "Acme" --prefix acme \
  --figma-name "Acme DS" --figma-key AbC123 \
  --scope @acme --author "Acme Inc" --repo https://github.com/acme/tokens \
  --dry-run
```

`--scope`, `--author`, and `--repo` are optional — include them if you have them,
leave them off if not.

### Step 2 — Run the scaffold for real
The same command, **without** `--dry-run`:

```bash
npm run scaffold-client -- \
  --name "Acme" --prefix acme \
  --figma-name "Acme DS" --figma-key AbC123 \
  --scope @acme --author "Acme Inc" --repo https://github.com/acme/tokens
```

This sets the client's name and token prefix, updates the package identity, and
regenerates the agent docs. (Exactly what it touches is in "What the scaffold
changes" below.)

### Step 3 — Install the tools and build
```bash
npm install     # downloads the project's tools (one-time; can take a few minutes)
npm run build   # turns your tokens into CSS/JS/JSON files inside dist/
```
If `npm run build` finishes without an error, the pipeline works. At this point
the token *values* are still IDEM's placeholders — Step 4 replaces them.

### Step 4 — Pull in the client's real tokens from Figma
Ask Claude in plain English:

> *"Sync tokens from the Acme Figma file into `tokens/*.json`, then run
> `npm run build`."*

Claude reads the client's Figma variables and rebuilds, so `dist/` now holds the
client's real colours, spacing, and type.

### Step 5 — Put the client under version control (git)
The new folder has no history yet (you left `.git` out in Step 0). Give it a fresh
one:

```bash
git init                                            # start tracking this folder
git add -A                                          # stage all the files
git commit -m "chore: scaffold Acme token pipeline" # save the first snapshot
```

To back it up on GitHub, create a new, empty repository at github.com (do **not**
tick "add a README"), copy its URL, then:

```bash
git remote add origin https://github.com/<your-username>/acme-tokens.git
git branch -M main         # name the main branch "main"
git push -u origin main    # upload
```

> From experience: if GitHub rejects the push with a message about `workflow`
> scope, it's because the commit changes a file under `.github/workflows/` and
> your access token isn't permitted to. Easiest fix: edit that file directly on
> github.com (which doesn't use the token), or grant your token the `workflow`
> permission in GitHub settings.

### Step 6 — Connect Chromatic (Storybook hosting)
In the new GitHub repo's **Settings → Secrets and variables → Actions**, add a
secret named `CHROMATIC_PROJECT_TOKEN` set to the project token from
chromatic.com. After that, each push redeploys Storybook automatically.

> One cosmetic item the scaffold doesn't change: Storybook titles/labels in
> `.storybook/*` and the story files may still say "IDEM" — adjust them if the
> client name appears in the Storybook UI.

---

### What the scaffold changes (reference)
When you run Step 2, `scripts/scaffold-client.mjs` does four things, in order:

1. **Rewrites `pipeline.config.mjs`** with the client's `projectName`, `prefix`, `figmaFileName`, `figmaFileKey`.
2. **Updates `package.json` identity** — `name` (`@acme/tokens` with `--scope`, else `acme-tokens`), resets `version` to `0.1.0`, sets `description`, and (if passed) `author` / `repository`. Also strips the hardcoded `--project-token=...` from the `chromatic` script so the CLI uses the `CHROMATIC_PROJECT_TOKEN` secret instead.
3. **Resets generated artifacts** — overwrites `tokens/snapshot.json` + `tokens/changelog.json` to a clean slate (`{}` / `[]`) and best-effort clears `dist/`. It resets by overwriting rather than deleting, so it works even on synced/sandboxed folders where file deletion is blocked; the next `npm run build` regenerates everything cleanly.
4. **Regenerates `AGENTS.md` + `CLAUDE.md`** from the template with the new values.

It deliberately **does not touch `tokens/*.json`**, so the repo still builds
immediately; those get replaced by the client's first Figma sync (Step 4). Use
`--dry-run` (Step 1) to preview every change without writing anything.

## Suggested end-state layout

```
@scope/token-pipeline-core   (published npm package)
├── sd.config.mjs            # prefix read from consumer config
├── scripts/snapshot-tokens.mjs
├── scripts/generate-docs.mjs
├── scripts/scaffold-client.mjs
├── templates/agent-rules.md # one source for AGENTS.md + CLAUDE.md
├── .storybook/
├── stories/                 # generic token stories
└── design.md, PROCESS.md    # shared rules

client-template              (GitHub template repo)
├── package.json             # depends on @scope/token-pipeline-core
├── pipeline.config.mjs      # { projectName, prefix, figmaFileName, figmaFileKey }
├── CLAUDE.md / AGENTS.md    # generated from templates/agent-rules.md
├── tokens/                  # filled by first Figma sync
└── .github/workflows/chromatic.yml

acme-tokens                  (an actual client repo)
└── template + `npm run scaffold-client` + their tokens/ + generated dist/
```

> The current repo keeps the core and template together (scripts + templates live
> alongside the tokens). Extract the core into its own package only once you've
> replicated a couple of clients and the machinery has stabilized.

## Production consumption

Each client repo's `dist/` is what production imports. Either publish that client
repo as its own package (`@acme/tokens`) and `import` the CSS/JS, or have the app
pull `dist/` on release. Production never depends on `IDEM`'s tokens — only on its
own client package.
